Question 8: 8. What is a 'NOP-sled' and how does it help an attacker exploit a stack buffer overflow?
Time Taken: 0.85s
-------------------- RESPONSE --------------------
A 'NOP-sled' is a technique used to exploit a stack buffer overflow. It involves corrupting a large section of the stack with no-op machine instructions, effectively increasing the size of the target area. At the end of the attacker-supplied data, after the no-op instructions, the attacker places an instruction to perform a relative jump to the top of the buffer where the shellcode is located. This collection of no-ops is referred to as the "NOP-sled" because if the return address is overwritten with any address within the no-op region of the buffer, it will "slide" down the no-ops until it is redirected to the actual malicious code by the jump at the end. This technique helps an attacker exploit a stack buffer overflow by solving the problem of finding the exact address of the buffer. [Slide 84.0]

Question 10: 10. Explain how a SYN Attack uses 'half-open connections' to saturate a server.
Time Taken: 0.93s
-------------------- RESPONSE --------------------
A SYN attack occurs when a host sends a flood of TCP/SYN packets, often with a forged sender address. Each of these packets is handled like a connection request, causing the server to spawn a half-open connection by sending back a TCP/SYN-ACK packet (Acknowledge) and waiting for a packet in response from the sender address (response to the ACK Packet). However, because the sender address is forged, the response never comes. These half-open connections saturate the number of available connections the server can make, keeping it from responding to legitimate requests until after the attack ends. This is a type of connection depletion attack that takes advantage of the fact that many connection-oriented protocols require servers to maintain state information after a connection request is made but before the connection is fully established. Question 13: 13. How does Blind SQL Injection differ from standard SQL Injection in terms of the feedback the attacker receives?
Time Taken: 0.85s
-------------------- RESPONSE --------------------
Blind SQL Injection differs from standard SQL Injection in that the attacker does not receive direct feedback from the database in the form of error messages or query results. Instead, the attacker must rely on indirect feedback, such as changes in the web application's behavior or content, to determine whether their injected SQL code was successful. This can make Blind SQL Injection more time-consuming and challenging, as the attacker must craft multiple queries to gather information about the database. [Slide 76.0]
